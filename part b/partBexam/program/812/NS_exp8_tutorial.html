Part A: Capturing and Analyzing Ethernet Frames

🧠 Concept Check: What is an Ethernet Frame?

In the OSI model, Layer 2 (Data Link Layer) deals with MAC addresses. When your computer sends data to your router, it packages the data into an "Ethernet Frame". This frame contains the physical MAC address of your computer (Source) and your router (Destination).

Step 1: Open Wireshark and Select Interface

When you open Wireshark, you will see a list of network interfaces (like Wi-Fi or Ethernet). Look for the one that has a squiggly line graph moving up and down—this means it has active internet traffic. Double-click that interface to start capturing.

Step 2: The Capture Screen & How to Stop

Immediately, your screen will fill up with hundreds of colorful rows. Let it run for 5 seconds while you open a website in your browser. Then, stop the capture.

Where to click: Click the ⬛ Red Square icon in the top left toolbar to stop capturing.

Step 4: Analyze the Packet

Click on any row in the top packet list (it will turn blue). Now look at the middle pane (the Packet Details pane).

Where to click: Look for the line that says Ethernet II. Click the small arrow > next to it to expand it.

You must write down these three things in your exam record:

  • Destination MAC Address: Where the frame is going (e.g., your router's hardware address).
  • Source MAC Address: Where it came from (your laptop's hardware address).
  • Type: Usually shows IPv4 (0x0800), proving that the Ethernet frame is carrying an IP packet inside it.

Part B: Simulate HTTP GET/POST Interaction

🧠 Concept Check: GET vs POST

GET: Used when you request data (like opening a webpage). Variables are sent in the URL.
POST: Used when you submit data (like a login form). Variables are hidden securely inside the body of the HTTP request.

Step 1: Capture HTTP GET

1. Start a new capture (click the blue shark fin). If it asks to save, click "Continue without saving".

2. Open Google Chrome and visit a non-secure website like http://example.com or http://neverssl.com. (Note: It MUST be HTTP, not HTTPS. HTTPS is encrypted and you won't see the text!)

3. Stop the capture.

Step 2: Filter and Analyze GET

Type the following in the filter bar:

http

Look for a packet in the list that says GET / HTTP/1.1. Click it.

Where to click: In the middle pane, expand Hypertext Transfer Protocol.

  • Notice the Method: GET
  • Right below it, find the response packet from the server. It will say HTTP/1.1 200 OK. This means the server successfully found the website.
Step 3: Capture HTTP POST

1. Start a new capture.

2. Open Chrome and go to a testing site like http://httpbin.org/forms/post.

3. Fill out the fake form with a dummy username/password and click Submit.

4. Stop the capture in Wireshark.

Apply this specific filter to find your POST request instantly:

http.request.method == "POST"

Where to click: Expand the HTTP section. Scroll to the very bottom of the middle pane and expand HTML Form URL Encoded. Here you will see the exact username and password you typed in plain text!

Part C: Simulate Capturing a Bulk TCP Transfer

🧠 Concept Check: The TCP Handshake

Before TCP sends a massive file, it must establish a connection using a 3-way handshake:
1. SYN: Client says "Hello, can we talk?"
2. SYN-ACK: Server says "Yes we can, here are my rules."
3. ACK: Client says "Understood, sending data now."

Step 1: Download a Large File

Start a capture. Go to your browser and download a relatively large file (e.g., a 50MB PDF or software installer). Stop the capture once the download finishes.

Step 2: Analyze the Handshake and Data

Type the following in the filter bar:

tcp

1. TCP 3-Way Handshake: Scroll to the top of your TCP packets. Look at the "Info" column for packets marked [SYN], [SYN, ACK], and [ACK].

2. Data Transfer: Look at the middle pane under Transmission Control Protocol. Observe the Sequence Numbers (Seq) and Acknowledgments (Ack). You will notice the Seq number increasing exactly by the size of the data sent!

3. Window Size: Look for the Window size value. This is Flow Control in action—it tells the sender how much buffer space the receiver has left.

Step 3: The Magic Trick - Follow TCP Stream

If you want to see the entire conversation assembled together, do this:

Where to click: Right-click on ANY of the TCP packets belonging to your download. In the menu that pops up, click Follow → TCP Stream.

A new window will pop up showing the raw data stream. Client requests will be in Red, and Server responses will be in Blue.

Step 4: Connection Termination

When the download is finished, TCP politely closes the connection. Scroll to the very bottom of your specific connection packets.

Look for the [FIN, ACK] flag. This stands for "Finish". The other side will reply with an [ACK], gracefully closing the socket.

🎉 You are now ready to ace Experiment 8!

Remember: Ethernet is Layer 2 (MAC). IP is Layer 3 (Routing). TCP is Layer 4 (Reliability). HTTP is Layer 7 (Application).
