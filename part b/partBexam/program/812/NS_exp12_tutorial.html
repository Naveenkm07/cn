Experiment 12: ICMP, PING & Traceroute Masterclass

Welcome to the comprehensive guide for understanding how the Internet tests itself! In this experiment, we will use the ICMP (Internet Control Message Protocol) to diagnose networks using two fundamental tools: ping and traceroute.

🧠 Core Concept: What is ICMP?

ICMP is the diagnostic protocol of the Internet (Layer 3). It doesn't carry user data (like HTTP or TCP does). Instead, it carries error messages and operational information. If a router is down, or a destination is unreachable, ICMP is the protocol that reports the problem back to the sender.

Part A: Analysis of ICMP and PING Messages

Step 1: Running the PING Command

Open your Command Prompt (Windows) or Terminal (Linux/Mac). To test if a server is alive, we send an "Echo Request".

Type ping google.com and press Enter.

Command Prompt
C:\Users\Student> ping google.com Pinging google.com [142.250.181.14] with 32 bytes of data: Reply from 142.250.181.14: bytes=32 time=20ms TTL=54 Reply from 142.250.181.14: bytes=32 time=21ms TTL=54 Reply from 142.250.181.14: bytes=32 time=20ms TTL=54 Reply from 142.250.181.14: bytes=32 time=19ms TTL=54 Ping statistics for 142.250.181.14: Packets: Sent = 4, Received = 4, Lost = 0 (0% loss), Approximate round trip times in milli-seconds: Minimum = 19ms, Maximum = 21ms, Average = 20ms
Step 2: Understanding the Output
  • TTL (Time to Live): Every time a packet passes through a router, TTL drops by 1. If TTL hits 0, the packet is destroyed (to prevent infinite loops in the network). Here, it arrived with 54 hops remaining.
  • Time (RTT - Round Trip Time): It took 20 milliseconds to travel to Google's server and come back to your computer.
  • Packet Loss: 0% means the network is highly reliable!

Part B: Analysis of ICMP and Traceroute Messages

🕵️‍♂️ Concept Check: The TTL Trick!

Traceroute is a genius hack. How do we find out the IP address of every router between us and Google?

We intentionally break packets by manipulating the TTL (Time to Live)!
- We send a ping to Google with TTL=1. The very 1st router receives it, drops TTL to 0, kills the packet, and angrily replies: "Hey! Time Exceeded!" (Boom, we just learned the 1st router's IP).
- Then we send a ping with TTL=2. It survives the 1st router, but the 2nd router kills it and replies. (Boom, we have the 2nd router's IP).
We keep increasing TTL until we finally hit Google!

Step 1: Running the Traceroute Command

Open your Command Prompt. Type tracert google.com (on Windows) or traceroute google.com (on Linux/Mac) and press Enter.

Command Prompt
C:\Users\Student> tracert google.com Tracing route to google.com [142.250.181.14] over a maximum of 30 hops: 1 <1 ms <1 ms <1 ms 192.168.1.1 (Your Home Router) 2 5 ms 4 ms 4 ms 10.10.10.1 (ISP Gateway) 3 10 ms 9 ms 9 ms 172.217.9.1 (Regional ISP Node) 4 12 ms 11 ms 12 ms 142.250.181.14 (Google Server) Trace complete.

Ultimate Comparison: PING vs Traceroute

Feature PING Traceroute
Primary Purpose Test if a single destination host is alive and reachable. Trace the exact path (every router) packets take to reach the destination.
ICMP Types Used Type 8 (Echo Request) & Type 0 (Echo Reply) Type 11 (Time Exceeded) & Type 0 (Echo Reply)
TTL Behavior Fixed TTL (e.g., starts at 64 or 128 and drops naturally). Artificially starts at TTL=1, then increases (TTL=2, TTL=3) on purpose!
What you see on screen Overall Round-trip time (RTT) and total packet loss %. List of every single hop (router) along the route with specific hop delays.
Best Use Case Basic connectivity check. "Is the server up?" Detailed network troubleshooting. "Where exactly is the network broken?"
